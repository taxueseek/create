# zaoren-content

> zaoren 系列·内容创作引擎。
> 把好内容拆解成四个步骤：看到什么、感受到什么、想明白了什么、可以怎么做。

---

## 在 zaoren 体系中的位置

本 skill 属于 **zaoren 系列** 中的「生产工具层」。

与系列成员的最佳协作方式：
- **选题/素材阶段** → 调用 `zaoren-roundtable` 让多角色碰撞灵感
- **内容生成阶段** → 用本 skill（`zaoren-content`）生成初稿
- **发布前风控** → 调用 `zaoren-creator-auditor`（网红）做选题/内容/变现/危机审计
- **风格校准** → 调用 `zaoren-reader` 从读者视角审视内容接受度

---

## 核心模型：四个步骤 × 黄金圈

所有好内容的底层结构都是一样的。不管是3分钟短视频还是3000字长文，不管是投资分析还是生活感悟，都在做同一件事：**把你看到的东西，变成别人能用的东西。**

我们把这件事拆成四个步骤，业内也叫「见感思行」：

| 步骤 | 核心问题 | 内容作用 | 深度 |
|----|----------|----------|------|
| **见（观察）** | 你看到了什么？ | 钩子、素材、现象描述 | 表层，人人能做 |
| **感（感受）** | 这让你有什么感觉？ | 共鸣、情绪连接、代入感 | 浅层，但威力巨大 |
| **思（思考）** | 你想明白了什么？ | 洞察、框架、认知升级 | 深层，区分高手和普通人 |
| **行（行动）** | 所以该怎么做？ | 方法、步骤、行动指引 | 落地，让内容有用 |

**关键洞察**：
- 只写「看到什么 + 感受什么」，就能出轻量内容（短视频、小红书笔记、朋友圈）
- 加上「想明白了什么」，变成有深度的观点输出
- 再加上「可以怎么做」，变成有价值的干货内容
- 四步全写，就是长文/深度视频的结构

**四个步骤与黄金圈的关系**：

黄金圈问 Why → How → What（为什么 → 怎么做 → 做什么）。
这四个步骤是从创作者的视角把黄金圈展开：

```
见（What）→ 你看到了什么现象/事实
感（Why 的入口）→ 为什么这个现象触动了你
思（Why 的深层）→ 底层逻辑是什么
行（How + What）→ 怎么做 + 具体做什么
```

## 创作流程

### 第一步：接收素材

用户给出的可能是：
- 一个观点（「信息差是伪命题」）
- 一个现象（「为什么AI课卖得好但没人真赚到钱」）
- 一段经历（「我今天看到一个排档生意火爆」）
- 一个话题（「资产配置」）
- 一个模糊的想法（「想讲讲关于选择的事」）

不管给的是什么，先做一件事：**用一句话提炼核心主张。**

如果用户给不出核心主张，用这四个步骤倒推：
- 你最初看到了什么？（见）
- 当时什么感觉？（感）
- 你从中想明白了什么？（思）
- 你觉得别人应该怎么做？（行）

四个问题走完，核心主张自然浮现。

### 第二步：定形态

| 形态 | 适用场景 | 用到哪几步 | 典型长度 |
|------|----------|-------------|----------|
| 短视频口播 | 一个观点讲透 | 见+感（轻量）或 见+感+思（中度） | 1-3分钟 |
| 小红书图文 | 经验分享、清单、对比 | 见+感+行（实用型）或 见+感（共鸣型） | 500-1000字 |
| 公众号长文 | 深度分析、系统思考 | 四步全用 | 2000-5000字 |
| 视频号/B站中视频 | 完整论述、故事+观点 | 四步全用 | 5-15分钟 |

如果用户没指定形态，根据内容深度自动推荐。一个素材可以同时出多种形态。

### 第三步：按四个步骤展开

**读取 `references/structure-guide.md` 获取每种形态的详细结构模板。**

## 四个步骤在不同形态中的展开

### 短视频口播（只写「见+感」的轻量版）

最小可发布单位。只要讲清楚「你看到了什么」和「你有什么感受」，就能出内容。

```
【见】你有没有注意到______？（现象/事实/场景）
【感】我第一反应是______。（真实感受，越具体越好）
      但后来我想，不对——______。（感受的转折）
【收】所以______。（一句话落点，不需要给方法论）
```

示例——话题「排档生意好」：

> 路过一家排档，晚上九点了还在排队。
> 我第一反应是：这老板真厉害。
> 但后来我想，不对。排队不是因为菜好吃，是因为他只有12张桌子。
> 供给决定了稀缺感。很多人以为是需求旺盛，其实是供给克制。

两步。没有道理，没有方法。但已经是一条完整的短视频。

### 短视频口播（加上「思」的中度版）

加一层「想明白了什么」，深度立刻上一个台阶。

```
【见】______（现象/事实）
【感】______（感受 + 感受的转折）
【思】这件事背后的逻辑是：______。（框架/原理/底层规律）
     大多数人以为是______，其实是______。
【收】______。（一句让人想截图的话）
```

### 公众号长文（四步全用）

```
【见·开篇】（500字以内）
  用一个具体的场景/现象/故事开头
  不要用「最近发现一个现象」，直接描述现象本身
  让读者在前三句就知道「这篇文章跟我有关」

【感·共鸣】（300字以内）
  说出读者对这个现象的感受——他自己说不清楚的那种
  「你可能也有这种感觉：______」
  这一层的任务是让读者觉得「被懂了」

【思·洞察】（1000-2000字，文章的核心）
  这是你真正要说的东西
  用框架/数据/案例/类比把道理讲透
  每个论点必须有一个「让人意外」的角度
  禁止罗列已知信息，必须有认知增量

【行·落地】（500字以内）
  给读者具体可以做的事
  不是「希望大家都能______」，是「明天你可以先做这一步：______」
  结尾不总结、不升华，给一个让人想一下的问题或一句够狠的话
```

### 小红书图文

```
标题：【见】层的一句话浓缩（必须有信息量，不能是「分享一个好方法」）

正文：
  【见】一段话描述场景/现象（2-3句）
  【感】我的感受/发现（1-2句）
  【行】具体方法/清单/步骤（主体，占70%篇幅）

  小红书的「思」层通常省略或融入「行」里
  用户来小红书是找答案的，不是来听你分析的
```

## 风格铁律

1. **第一句话决定生死**。不管什么形态，第一句必须让人留下来。不自我介绍、不铺垫背景、不说「今天跟大家聊聊」。
2. **禁止正确的废话**。「投资有风险」「每个人情况不同」「需要具体问题具体分析」——这些话没有信息量，砍掉。
3. **具体胜过抽象**。不说「效率很重要」，说「我用这个方法每天多出来两个小时」。
4. **一篇内容只说一件事**。想说的太多就拆成系列，不要往一篇里塞。
5. **每篇内容必须有一句「截图级」的话**。读完之后让人想存下来或者发给别人的那一句。找不到就重写。
6. **说完就停**。不总结、不升华、不「希望对大家有帮助」。
7. 使用直角引号「」，少用加粗，不用破折号。

## 素材到多形态一鱼多吃

一个素材可以同时产出多种形态：

```
原始素材：一个观点/经历/发现

→ 短视频口播（见+感，1分钟）：只讲现象和感受，留悬念
→ 小红书图文（见+感+行）：加上方法论，做成可收藏的清单
→ 公众号长文（见+感+思+行）：完整展开，深度分析
→ 朋友圈文案（见+感，3句话）：最浓缩的版本
```

**执行规则**：用户给一个素材，如果没指定形态，默认先出短视频口播版本（最小可发布单位），然后问是否需要扩展到其他形态。

## 选题诊断

如果用户不知道写什么，用这四个步骤做选题诊断：

```
最近你看到了什么让你「停下来想了一下」的事？（见）
  ↓
你当时是什么感觉？生气？惊讶？觉得哪里不对？（感）
  ↓
你后来想明白了什么？或者还没想明白？（思）
  ↓
你觉得别人应该知道这件事吗？为什么？（行的种子）
```

能回答前两个问题，就有一条短视频。能回答四个，就有一篇长文。

## 输出前自检

1. **见层检验**：开头的「见」够不够具体？能不能让人脑子里出现画面？出不了画面就重写。
2. **感层检验**：感受是真实的还是装出来的？「我觉得很有启发」是装的，「我当时第一反应是这不可能」是真的。
3. **思层检验**：洞察有没有认知增量？读者看完会不会觉得「原来是这样」？如果只是常识，不算洞察。
4. **行层检验**：行动建议够不够具体？「你也应该多思考」不算，「明天起床后花5分钟写三个问题」才算。
5. **截图级检验**：整篇内容里有没有一句话值得截图？没有就重写到有。

## 发布后风控（必选动作）

内容生成后，在发布前**必须建议用户调用 `zaoren-creator-auditor`（网红）做最后一轮安全审计**。这是创作流程的闭环：

```
生产（zaoren-content）→ 风控（zaoren-creator-auditor）→ 发布
```

## 与 zaoren 生态的协作

- 用 `zaoren` 造出来的行业 skill（如 `zaoren-investor`、`zaoren-lawyer`），可为创作提供专业判断力和权威素材

- 用 `zaoren-creator-auditor` 在发布前拦截风险
- 用 `zaoren-reader` 从读者视角校准内容接受度
- 用 `trust-protocol` 管理内容合作中的信任博弈（甲方谈判、商务边界）
