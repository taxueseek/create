# 分析指南与正文模板

本文件在两种情况下读取：
1. Step 1遇到「结构不清晰」或「复合型」材料，需要深度分析
2. Step 5写正文时，需要对应类型的模板

---

## 复杂材料识别方法

### 「步骤」还是「规则」？

元提示词经常把规则伪装成步骤：
```
第一步：请按照以下格式输出...（格式规范，不是步骤）
第二步：注意以下几点...（约束，不是步骤）
第三步：生成内容...（真正的步骤）
```

判断方法：真正的步骤会改变「任务状态」；只是描述「怎么做」的是规则。
处理：规则归入「核心约束」章节，不列为并列步骤。

### SOP文档的隐含结构

很多SOP写的是「操作动作」，但隐含了「判断逻辑」没有写出来。
转化时需要把隐含的判断逻辑显式化，否则skill执行时会流于形式。

例：「选择合适的基金」（流程型写法）
→ 实际隐含：「按什么维度筛选、每个维度的阈值是多少」（判断型）
→ 转化时应提取判断框架，不只是保留操作步骤。

### 书/笔记的可执行化潜力

判断一本书的笔记是否值得做skill，看三点：
- 书里有没有可操作的框架/清单/决策树？（有→判断型或流程型）
- 书的核心是「怎么思考」还是「知道什么」？（前者→判断型，后者→知识型）
- 你是否会反复用这本书的框架处理新问题？（会→做skill，不会→做文档）

## 各类型正文模板

### 流程型模板

```markdown
## Step N：[步骤名]

[AI的行动，1-3句，说清楚做什么、依据什么]

> 需要详细规范时：参考 `references/xxx.md`

**输出格式**：
[具体的输出结构描述或代码块示例]

**用户确认**：[如果这一步需要用户决策，说明确认什么]
```

### 判断型模板

```markdown
## 判断框架

### 维度1：[名称]
[判断标准，具体到可操作——有数字/有对比/有条件]

**好的例子**：[一句话描述什么情况算好]
**差的例子**：[一句话描述什么情况算差]

> 典型案例库：`references/cases.md`

### 综合判断逻辑
[多个维度如何加权/如何取舍，有先后顺序就写先后]
```

### 风格型模板

```markdown
## 风格DNA

### 永远遵守
- [规则1：具体到可以验证]
- [规则2]

### 永远避免
- [反规则1：说清楚为什么避免，不只是「不要用」]
- [反规则2]

### 标志性表达
[3-5个典型句式或结构，直接可以仿写]

> 完整示例库：`references/examples.md`
```

### 知识库型模板

```markdown
## 使用方式

这个skill是[领域]的知识索引。用户提问时，先判断问题类型，
再读取对应的references文件，基于其中的知识回答。

## 问题类型索引

| 用户问的是... | 读取哪个文件 |
|-------------|------------|
| [问题类型A] | `references/topic-a.md` |
| [问题类型B] | `references/topic-b.md` |
| 综合性问题   | 先读index，再按需读具体文件 |

## 回答原则
[基于这个知识库回答时的特殊要求，比如：引用来源/区分确定与推断/等]
```

### 思维模型型模板

```markdown
## 触发条件

在以下情境下调用本框架：
- [情境1：用一句话描述什么样的问题应该用这个框架]
- [情境2]
- [情境3]

不适用情境：[说清楚什么情况下套用会失效]

## 框架结构

[框架名称]的操作步骤：

1. [第一步：做什么，问自己什么问题]
2. [第二步]
3. [第三步]

核心变量：[框架里最关键的判断维度，2-4个]

> 框架细节与边界说明：`references/frameworks.md`

## 调用姿态

使用此框架时，Claude应当：
- [具体的思维方式要求，比如：先假设反面成立再论证/从最小单元开始/等]
- [什么时候停止，输出什么]

> 真实案例库：`references/cases.md`
```

## 分层决策树

```
这段内容放哪里？
│
├── 去掉它skill还能运行？
│   ├── 不能 → 骨架，留SKILL.md
│   └── 能 → 继续判断↓
│
├── 每次执行都一样？
│   ├── 是，体量小（<20行）→ 留SKILL.md
│   ├── 是，体量大（>20行）→ 移references
│   └── 否，随用户输入变化 → 运行时传入，不写进文件
│
└── 是「半动态」（选项固定，用户选不同）？
    → 选项清单放references，选择逻辑留SKILL.md
```

## description写作模板

```
[skill做什么，一句话]。触发条件：用户[触发场景1]、[触发场景2]、[触发场景3]，
或者用户[模糊触发场景]。即使用户只说「[最模糊的触发表达]」也应触发。
[可选：补充说明核心能力或输出形态]。
```

**易犯错误**：触发描述只写「正式」的触发词，漏掉用户的模糊/口语表达。
比如「转化元提示词」是正式触发，「我每次都要重复做这件事」才是模糊触发，两个都要写。

## 常见元提示词模式对照

| 原始特征 | 转化方式 |
|---------|---------|
| 「第一步分析X，第二步选Y，第三步生成Z」 | 流程型，Y选项放references |
| 「以下格式输出，每项包含A/B/C」 | references/templates.md存放格式模板 |
| 「从以下10种风格中选1种」 | references/options.md存放候选项 |
| 「必须包含/不得出现」类强制规则 | SKILL.md核心约束章节 |
| 批量生成（如10张海报） | references按类型拆分 |
| 「如果上传图片则X，否则Y」 | SKILL.md写条件分支，两路径都写 |
| 「🔴重中之重」标注 | 提炼为核心约束清单，动词开头 |
