# 知识 → Skill 转化引擎

一切已经结晶的认知，只要满足「可重复触发」+「有明确输出」，都能变成skill。

---

## 第一原则：什么值得变成skill

在动手之前，先回答这个问题：

> 如果把这份知识变成skill，Claude调用它时能做出「没有它就做不到的事」吗？

| 条件 | 结论 |
|------|------|
| 重复触发 + 输出可验证 | ✅ 适合做skill |
| 重复触发 + 输出主观 | ✅ 适合做persona/style skill |
| 一次性 + 输出可验证 | ⚠️ 考虑做模板，不一定要做skill |
| 纯叙事/背景知识 | ❌ 做参考文档，嵌入其他skill的references |

---

## Step 1：识别知识类型

拿到任何输入材料，先判断类型，因为不同类型对应不同skill架构。

> 复杂材料的识别方法见 `references/analysis-guide.md`

**A. 流程型**
特征：有明确步骤顺序，每步有输入输出，可以被照着执行。
来源：SOP、操作手册、元提示词的工作流部分、方法论文章。
架构：以工作流步骤为主干，选项/模板放references。

**B. 判断型**
特征：核心是「在什么情况下做什么决策」，有判断框架和标准。
来源：投资分析框架、选题标准、Review原则、风险评估清单。
架构：以判断树/决策矩阵为核心，案例放references。

**C. 风格型**
特征：核心是「怎么写/怎么表达」，有语气、结构、用词偏好。
来源：个人写作风格、品牌文案规范、编辑标准、某作者的笔记风格。
架构：以风格规则+反例为核心，大量示例放references。

**D. 知识型**
特征：大量领域知识，但没有明显流程或判断框架。
来源：一本书的笔记、研究报告、行业综述。
处理：不直接做skill，而做「知识检索型skill」——引导Claude如何从知识库里找答案，而不是把知识本身塞进skill。
⚠️ 注意：如果输入是某人的大量原始语料（文章/播客/公众号存档），**先用 knowledge-extractor 提炼**，再把提炼结果带回来做E类处理。原始语料直接转化会丢失隐性框架。

**E. 思维模型型**
特征：已经过提炼的知识资产——隐性框架、核心信念、决策模式，通常来自 knowledge-extractor 的输出。
来源：knowledge-extractor 的 Phase 3-4 输出、个人反复验证过的判断逻辑、某领域专家的分析结构。
架构：以「触发条件 + 调用方式」为核心——skill 的职责不是存储知识，而是告诉 Claude 在什么情境下调用哪个框架、以什么姿态思考。references 存放框架细节和示例。
与判断型的区别：判断型处理「具体决策的标准」，思维模型型处理「思考问题的方式本身」。

**F. 复合型**
特征：兼具以上多种。
处理：识别主类型，其余降级为references。

识别完成后告知用户：「这份材料是[类型]，建议做成[架构描述]的skill，核心是[X]，references存放[Y]。」

## Step 2：提取可执行核心

**流程型** → 提取「步骤骨架」
- 找出真正改变任务状态的步骤（不是规则伪装成的步骤）
- 识别每步的输入条件和输出形态
- 标记用户决策点

**判断型** → 提取「判断框架」
- 找出核心判断维度（通常3-7个）
- 识别每个维度的评判标准
- 收集典型好/坏案例作为参照

**风格型** → 提取「风格DNA」
- 找出不可违反的风格规则（「永远不用」「必须出现」）
- 识别标志性句式/结构/用词
- 收集典型正例和反例

**知识型** → 提取「检索框架」
- 整理知识的分类结构
- 识别用户最可能问的问题类型
- 建立「问题类型 → 知识位置」的索引

**思维模型型** → 提取「调用协议」
- 识别框架的触发情境（什么问题/场景下该用这个框架）
- 提炼框架的操作步骤（用这个框架时，思考顺序是什么）
- 标记框架的适用边界（什么情况下不该套用）
- 收集 2-3 个框架处理真实问题的案例，放 references

## Step 3：设计分层结构

**分层的唯一标准**：

> 去掉这段内容，skill还能运行吗？
> - 能运行 → 它是参数，放references或运行时传入
> - 不能运行 → 它是骨架，留在SKILL.md

三层结构：

```
SKILL.md正文        骨架层：工作流/判断框架/风格DNA
references/         知识层：选项列表/模板/案例/详细规范
运行时输入          动态层：用户每次提供的具体内容
```

「半动态」内容的处理：候选项固定但用户每次选不同——候选项清单放references，「选择逻辑」留SKILL.md，用户的具体选择在运行时确认。

体量控制：SKILL.md目标200-300行，上限400行；单个reference文件200-400行；超出就继续拆。

## Step 4：选择文件结构

**极简型**（单一流程，规则少）
```
skill-name/
└── SKILL.md
```

**标准型**（最常见）
```
skill-name/
├── SKILL.md
└── references/
    ├── options.md
    └── templates.md
```

**领域分支型**（同一工作流在不同领域差异大）
```
skill-name/
├── SKILL.md
└── references/
    ├── domain-a.md
    ├── domain-b.md
    └── shared-rules.md
```

**批量输出型**（生成多个结构类似的产物）
```
skill-name/
├── SKILL.md
└── references/
    ├── unit-templates.md
    └── style-options.md
```

**知识库型**（D类输入）
```
skill-name/
├── SKILL.md                 # 检索框架 + 问答引导逻辑
└── references/
    ├── index.md             # 问题类型 → 知识位置索引
    ├── topic-a.md
    └── topic-b.md
```

**思维模型型**（E类输入）
```
skill-name/
├── SKILL.md                 # 触发条件 + 调用协议 + 适用边界
└── references/
    ├── frameworks.md        # 框架细节与操作步骤
    └── cases.md             # 真实案例库
```

## Step 5：写 SKILL.md

**frontmatter**：

```yaml
---
name: skill-name
description: [做什么，一句话]。触发条件：[3-5个用户可能说的短语]。
即使用户只说「[最模糊的触发表达]」也应触发。[输出形态补充说明]。
---
```

description是触发机制，80-120字，宁啰嗦也要覆盖模糊表达。

**正文结构按类型选择**，见 `references/analysis-guide.md` 中的模板。

**核心约束章节**（所有类型都需要）：
把原始材料里「必须/不得/重中之重」提炼为编号清单，动词开头，每条一行，最多10条，越具体越好——「保持风格一致」不如「所有标题用无衬线字体，正文用衬线字体」。

---

## Step 6：自检

- [ ] description覆盖了模糊触发表达
- [ ] SKILL.md里没有大段候选项（>8条的列表应在references）
- [ ] 每个reference文件开头说明「何时读/包含什么」
- [ ] 动态输入没有硬编码
- [ ] 核心约束是具体可操作的，不是抽象原则
- [ ] 去掉SKILL.md任何一段，skill是否仍能运行？（能运行就移走）

## 本 skill 的边界

做：把已有知识转化为可执行 skill
不做：从零设计全新 skill（用 skill-creator）
不做：评估 quality、运行测试、迭代优化（用 skill-creator）
不做：执行 skill 描述的工作（安装后直接触发对应 skill）
不做：轻量个人工具（用 `taxue-build`，见下文）

**与 taxue-build 的分工**

| 需求 | 选择 | 原因 |
|------|------|------|
| 个人模板、SOP、检查清单，15分钟完成 | `taxue-build` | 轻量快速，自用为主 |
| 多角色认知系统、完整 Skill、需要世界观设计 | `zaoren-compile` | 需要设计角色、盲区、第一性原理 |
| 不确定选哪个 | 先 `taxue-build`，复杂了再升级 | 从简单开始 |

**与 knowledge-extractor 的协作关系**：

```
原始大语料（文章/书/播客）
    ↓ knowledge-extractor
已提炼的知识资产（框架/信念/洞察）
    ↓ knowledge-to-skill（E类：思维模型型）
可安装的 skill 文件
```

如果用户拿着原始语料直接来这里——先建议他跑一遍 knowledge-extractor，
拿到 Phase 3-4 的输出再回来。直接转化会丢失隐性框架，只能做到 D 类知识检索，
而不是 E 类思维模型。
