# 需求诊断：四条造物路径

用户进来时，不急着造什么。先判断哪条路径最高效。

---

## 为什么需要路径选择

「造一个角色」只是造物的方式之一，不是唯一方式，很多时候甚至不是最高效的方式。

角色路径的隐性假设是：用户知道自己需要什么角色。但很多时候用户知道的是「我在哪里反复出问题」「这件事最容易死在哪里」「我需要有人在关键时刻替我把关」——这些需求不一定要包装成一个「人」。

四条路径不是互相替代的。它们是不同的入口，最终都产出可安装的认知引擎。区别在于从哪里切入最准。

---

## 路径一：角色路径（现有能力）

**入口信号**：用户明确知道要造什么。
- 「造我老板」「造一个产品经理」「帮我搭个投研团队」
- 用户有明确的角色概念，或有具体素材想提取

**执行方式**：直接进入六大造物类型（人物/职业/智慧/行业/项目/团队）。

**适用场景**：用户对自己的需求有清晰认知，知道要什么角色、要什么能力。

---

## 路径二：决策链路径

**入口信号**：用户描述的是一个**流程或场景**，而不是一个角色。
- 「我每次选基金都不知道先看什么」
- 「从接到需求到上线，中间老是出问题」
- 「我的投资流程需要优化」
- 「帮我理一下从选题到发布的流程」

**核心逻辑**：不造角色，造**决策流水线**。

**执行步骤**：

### 1. 提取决策链

把用户的场景拆成一串决策节点：

```
触发事件
  ↓
决策节点1：______？
  ├── 通过 → 进入节点2
  └── 不通过 → 执行______（终止/回退/转向）
  ↓
决策节点2：______？
  ├── 通过 → 进入节点3
  └── 不通过 → 执行______
  ↓
...
  ↓
最终行动：______
```

### 2. 为每个节点定义判断系统

每个决策节点需要明确：

| 要素 | 内容 |
|------|------|
| 输入信息 | 做这个判断需要看什么数据/事实 |
| 判断标准 | 什么情况通过，什么情况不通过（必须具体、可量化） |
| 不通过后果 | 不通过意味着什么，下一步做什么 |
| 常见误判 | 这个节点最容易犯什么错 |
| 耗时预期 | 这个判断应该花多长时间（防止过度分析） |

### 3. 标注关键拦截点

决策链中有些节点是「闸门」——过了这个点就很难回头。在这些节点上加重判断力度，增加交叉验证。

### 4. 输出格式

```
zaoren-chain-{场景名}/
├── SKILL.md              # 入口：决策链概述 + 触发条件
├── chain.md              # 完整决策链（流程图 + 节点详情）
├── gates/                # 关键闸门的深度判断规则
│   ├── gate-1.md
│   └── gate-2.md
└── failure-log.md        # 常见误判和历史教训
```

### 运行方式

用户遇到触发事件时，skill 自动从头走决策链，逐个节点给出判断建议。用户可以在任何节点停下来追问「为什么这么判断」或「如果我不同意呢」。

**示例：选基金决策链**

```
触发：发现一只感兴趣的基金

节点1：基金经理在任超过3年吗？
  ├── 是 → 节点2
  └── 否 → 放弃（管理时间太短，无法验证能力）

节点2：近3年最大回撤超过同类平均吗？
  ├── 没超过 → 节点3
  └── 超过 → 标记风险，进入节点3但降低仓位上限

节点3：持仓集中度前十大持仓占比超60%吗？
  ├── 没超过 → 节点4
  └── 超过 → 需要判断集中持仓的逻辑是否成立

节点4：费率在同类中处于什么位置？
  ├── 低于中位数 → 节点5
  └── 高于中位数 → 追问：高费率换来了什么

节点5：与现有持仓的相关性
  ├── 低相关 → 通过，进入建仓决策
  └── 高相关 → 重新评估是否重复暴露
```

## 路径三：失败模式路径

**入口信号**：用户描述的是**问题、痛点、反复犯的错**。
- 「我炒股总是追高」
- 「我的餐厅老是留不住客人」
- 「写的文章数据总是很差」
- 「项目总是延期」
- 「我每次都在XX环节栽跟头」

**核心逻辑**：不问「你需要谁」，问「你最常怎么死」。从失败模式反推需要的判断能力。

**执行步骤**：

### 1. 失败模式诊断

引导用户列出反复出现的问题，然后归类：

```
用户描述的问题
  ↓
归类为失败模式（通常3到5种）
  ↓
每种失败模式的触发条件是什么？
  ↓
在什么环节可以提前拦截？
```

### 2. 为每种失败模式建立拦截系统

| 要素 | 内容 |
|------|------|
| 失败模式名称 | 一句话描述（如：「追高」「过度集中」「忽视成本」） |
| 触发条件 | 什么情况下容易犯这个错（具体、可识别） |
| 早期信号 | 在犯错之前有什么征兆 |
| 拦截规则 | 出现信号时自动执行什么（硬性拦截，不是建议） |
| 应对预案 | 如果已经犯了，怎么止损 |
| 历史案例 | 过去犯这个错的具体经历（如果用户提供了素材） |

### 3. 组装成防御系统

不是一个角色，是一组**拦截规则**。在用户的决策流程中，嵌入在最容易出错的环节。

### 4. 输出格式

```
zaoren-shield-{场景名}/
├── SKILL.md              # 入口：防御系统概述
├── failure-modes.md      # 失败模式全景图
├── interceptors/         # 每种失败模式的拦截规则
│   ├── mode-1.md
│   ├── mode-2.md
│   └── mode-3.md
└── recovery.md           # 已经犯错后的止损预案
```

### 运行方式

用户在做决策时，skill 自动扫描是否触发了某种失败模式的早期信号。如果触发，直接弹出警告和拦截建议，不等用户主动问。

**示例：投资防御系统**

```
失败模式1：追高
  触发条件：标的近1个月涨幅超过20%，且用户此时想买入
  拦截规则：强制冷静期24小时，期间必须回答「如果明天跌10%你能接受吗」
  
失败模式2：过度集中
  触发条件：单一标的仓位即将超过总仓位的20%
  拦截规则：硬性阻断，必须先减仓其他标的或说明集中持仓理由
  
失败模式3：不止损
  触发条件：持仓浮亏超过预设止损线
  拦截规则：自动触发止损评估流程，不执行则每日提醒
```

## 路径四：行为审计路径

**入口信号**：用户有**历史记录**可以分析。
- 「这是我过去一年的交易记录」
- 「这是我写过的50篇文章的数据」
- 「这是我项目的决策日志」
- 「帮我看看我的操作有什么问题」

**核心逻辑**：从用户的真实行为中提取「你实际上是怎么做决策的」，找出系统性偏差，在最薄弱的环节补判断系统。

**执行步骤**：

### 1. 行为模式提取

分析用户的历史记录，提取：
- **实际决策模式**：你在做X的时候，真正的决策流程是什么（而不是你以为的流程）
- **系统性偏差**：哪些错误反复出现，概率多高
- **能力边界**：哪些判断你做得很好，哪些你做得很差
- **时间模式**：什么时间段/什么情绪状态下容易犯错

### 2. 能力地图

```
用户能力地图：

强项（自己做比AI好）：
  - ______
  - ______

弱项（需要AI补位）：
  - ______（补位方式：______）
  - ______（补位方式：______）

盲区（自己不知道自己不知道）：
  - ______（发现来源：______）
```

### 3. 定制补位系统

不是给用户一个完整的团队替他干活，而是精准识别他最薄弱的环节，只在那里插入判断系统。

**原则**：用户做得好的地方不干预。只在用户做不好的地方介入，而且介入方式不是替代，是**提醒和拦截**。

### 4. 输出格式

```
zaoren-audit-{用户名}/
├── SKILL.md              # 入口：审计结论 + 补位策略
├── behavior-map.md       # 行为模式分析
├── capability-map.md     # 能力地图（强项/弱项/盲区）
├── patches/              # 补位系统（只针对弱项和盲区）
│   ├── patch-1.md
│   └── patch-2.md
└── evolution-log.md      # 能力成长追踪
```

### 运行方式

初次使用：分析历史数据，输出能力地图和补位建议。
日常使用：在用户决策过程中，只在识别到弱项/盲区相关决策时介入。
长期进化：定期重新审计，追踪哪些弱项已经改善（可以撤掉补位），哪些新盲区浮现。

## 路径选择路由

当用户的需求不能直接判断类型时，用以下诊断问题路由：

```
用户进来了
  ↓
Q1：你有没有明确想造的角色/团队？
  ├── 有 → 路径一（角色路径），进入六大类型
  └── 没有 → Q2
  ↓
Q2：你有没有一个需要优化的流程？
  ├── 有 → 路径二（决策链路径）
  └── 没有 → Q3
  ↓
Q3：你有没有反复出现的问题或经常犯的错？
  ├── 有 → 路径三（失败模式路径）
  └── 没有 → Q4
  ↓
Q4：你有没有历史数据/记录可以分析？
  ├── 有 → 路径四（行为审计路径）
  └── 没有 → 回到路径一，用引导式问答帮用户明确需求
```

**注意**：路径之间可以串联。比如用户走了路径四（行为审计），发现自己在「选基金」环节反复犯错，然后切入路径二（决策链）专门优化这个环节。或者走了路径三（失败模式），发现需要一个风控角色来执行拦截规则，然后切入路径一造一个专属风控官。

---

## 路径产出对比

| 路径 | 输入 | 核心产出 | 适合谁 |
|------|------|----------|--------|
| 角色路径 | 素材或场景描述 | 可安装的角色/团队 | 知道自己要什么的用户 |
| 决策链路径 | 流程描述 | 决策流水线 | 知道做什么但不知道怎么做的用户 |
| 失败模式路径 | 痛点描述 | 防御拦截系统 | 知道自己哪里出问题的用户 |
| 行为审计路径 | 历史记录 | 能力地图 + 精准补位 | 有数据但不知道问题在哪的用户 |

**四条路径，同一个目的地：让用户的判断力系统性提升。**
