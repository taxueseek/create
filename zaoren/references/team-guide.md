# 造团队：组织蓝图生成引擎

本文件定义「造团队」的完整方法论。由 SKILL.md 在识别到团队类型需求时调用。

---

## 核心逻辑

造单个角色是造「一个脑子」。造团队是造「一个组织的神经系统」——谁负责感知、谁负责判断、谁负责执行、谁负责纠错，以及他们之间怎么传递信息、怎么产生冲突、怎么收敛决策。

一个团队不是几个角色的简单拼凑。角色之间必须有：
- **分工边界**：谁管什么，不能重叠也不能有真空
- **汇报关系**：信息往哪里流，决策权在谁手里
- **内置张力**：哪些角色之间天然有利益冲突，这种冲突怎么变成质量保障而不是内耗
- **协作协议**：日常怎么配合，异常怎么升级

## 团队生成五步法

### 第一步：场景拆解

用户给出场景后（「我是港A股个人投资者」「我要开一个排档」），先拆解场景的**价值链**：

**核心问题**：这件事从开始到结束，必须经过哪些环节？每个环节的核心能力是什么？

示例——港A股个人投资者：
```
信息采集 → 研究分析 → 投资决策 → 组合管理 → 风险控制 → 复盘迭代
```

示例——排档创业：
```
选址定位 → 菜单设计 → 供应链 → 厨房生产 → 前厅服务 → 财务管控 → 营销获客
```

### 第二步：角色推导

从价值链的每个环节反推：**这个环节需要什么样的判断能力？**

推导规则：
1. 每个核心环节至少对应一个角色
2. 相邻且能力相近的环节可以合并（一人多岗）
3. 用户本人（老板/决策者）必须是团队的一个显式角色，明确其职责边界
4. 团队总人数控制在3到7个（含用户自身）

推导时使用**角色必要性三问**：
- 去掉这个角色，哪个环节会失控？（如果没有环节失控，这个角色是多余的）
- 这个角色的判断，用户自己能做吗？（如果能，合并到用户角色里）
- 这个角色跟哪个角色天然有冲突？（如果跟谁都没冲突，可能是万金油，要重新定义）

### 第三步：组织架构设计

确定角色之间的关系结构：

**架构三要素**：

1. **决策层级**
```
老板（用户）
├── 核心岗1（有独立决策权的范围）
├── 核心岗2（有独立决策权的范围）
└── 支撑岗（执行层，向核心岗汇报）
```

2. **信息流向**
- 哪些信息必须上报给用户（决策级）
- 哪些信息角色之间横向传递（协作级）
- 哪些信息角色自行处理（执行级）

3. **内置张力**
- 标注哪些角色之间天然存在利益冲突
- 说明这种冲突为什么是健康的（相互制衡）
- 定义冲突升级机制（什么情况下提交给用户裁决）

### 第四步：逐角色生成

对每个角色，使用造人.skill的通用提取框架生成完整skill。但团队角色比独立角色多一层：**组织行为层**。

每个角色的输出结构：

```
【{角色名}：{职位}】

一句话定位：______

职责边界：
  管什么：______
  不管什么：______
  灰色地带（需要跟谁协商）：______

决策权限：
  独立决策：______（不需要请示老板的事）
  建议权：______（必须给建议但老板拍板的事）
  执行权：______（老板决定后负责执行的事）

判断框架：
  1. 先看______
  2. 再看______
  3. 红线：______

汇报对象：______
协作对象：______
天然冲突对象：______（冲突原因：______）

表达风格：______
```

### 第五步：团队运行手册

生成整个团队的协作规范：

```
团队运行手册

【日常协作协议】
- 信息同步：______（谁在什么时间向谁同步什么）
- 交叉检查：______（谁检查谁的输出）
- 例会机制：______（什么频率、什么议题、谁参加）

【决策升级协议】
- Level 1（角色自行决策）：______
- Level 2（需要协商）：______
- Level 3（必须提交老板）：______

【异常处理协议】
- 出现______情况时，由______牵头处理
- 出现______情况时，全员暂停，老板决策

【团队自检清单】
每周/每月自检：
□ 每个角色是否在其职责范围内运行？
□ 是否有环节没人管（真空）？
□ 是否有环节多人管（内耗）？
□ 内置张力是否在产生正向制衡？
```

## 场景模板库

以下是高频场景的角色推导示例，用于加速团队生成。用户不给素材时直接基于模板生成。

### 模板一：个人投资者（一人公司）

**价值链**：信息采集 → 研究分析 → 投资决策 → 组合管理 → 风险控制 → 复盘迭代

**角色配置**：

| 角色 | 一句话定位 | 独立决策范围 |
|------|-----------|-------------|
| 老板（用户） | 最终决策者，定战略方向 | 买卖决策、仓位上限、策略切换 |
| 研究员 | 挖信息、出报告、给判断 | 研究选题、分析框架选择 |
| 基金经理 | 管组合、控节奏、算账 | 仓位调整（在老板定的范围内）、再平衡时机 |
| 风控官 | 找风险、设红线、踩刹车 | 触发止损、否决超限操作 |

**内置张力**：
- 研究员 vs 风控官：研究员倾向发现机会，风控官倾向规避风险。研究员说「这个值得买」，风控官说「你算过最大回撤吗」。
- 基金经理 vs 研究员：基金经理关注组合整体平衡，研究员关注单个标的深度。基金经理说「你的推荐跟组合里已有持仓相关性太高」。

### 模板二：餐饮创业（排档/小店）

**价值链**：选址定位 → 菜单设计 → 供应链 → 厨房生产 → 前厅服务 → 财务管控

**角色配置**：

| 角色 | 一句话定位 | 独立决策范围 |
|------|-----------|-------------|
| 老板（用户） | 定方向、管钱、拍板 | 选址、定价策略、大额采购、扩张决策 |
| 厨师长 | 管出品、控品质、调菜单 | 菜品研发、备料标准、出餐流程 |
| 领班 | 管人、管场、管客诉 | 排班、现场应急、服务流程 |
| 采购/财务 | 管成本、管供应商、算账 | 日常采购（预算内）、供应商比价 |

**内置张力**：
- 厨师长 vs 采购：厨师长要好食材，采购要控成本。厨师长说「这个供应商的鱼不够新鲜」，采购说「换供应商每月多花3000」。
- 领班 vs 厨师长：领班代表客户体验，厨师长代表出品标准。领班说「客人等太久了」，厨师长说「不能为了快砍工序」。

### 模板三：内容创业（自媒体/知识付费）

**价值链**：选题策划 → 内容生产 → 分发运营 → 用户运营 → 商业变现

**角色配置**：

| 角色 | 一句话定位 | 独立决策范围 |
|------|-----------|-------------|
| 老板（用户） | 内容方向、品牌调性、变现策略 | 选题方向、合作决策、定价 |
| 编辑/内容官 | 质量把关、结构优化、风格校准 | 稿件修改、标题选择、排版规范 |
| 运营官 | 数据驱动、平台适配、增长策略 | 发布时间、平台选择、互动策略 |
| 商务/变现 | 找钱路径、客户筛选、价值定价 | 广告筛选（符合调性前提下）、社群运营 |

**内置张力**：
- 编辑 vs 运营：编辑追求内容深度，运营追求传播广度。编辑说「这篇文章需要3000字才能讲清楚」，运营说「超过1500字完读率掉一半」。
- 老板 vs 商务：老板守品牌调性，商务找变现机会。商务说「这个广告主愿意出5万」，老板说「跟我们调性不符」。

### 模板四：技术创业（小型开发团队）

**价值链**：需求定义 → 技术设计 → 开发实现 → 测试验收 → 上线运维 → 用户反馈

**角色配置**：

| 角色 | 一句话定位 | 独立决策范围 |
|------|-----------|-------------|
| 老板/产品（用户） | 需求优先级、产品方向、资源分配 | 做什么不做什么、版本规划 |
| 架构师 | 技术选型、系统设计、技术债管控 | 技术方案选择、重构时机 |
| 开发主力 | 写代码、估工期、交付质量 | 实现方式、开发工具选择 |
| QA/用户代言人 | 找bug、代表用户、质量兜底 | 测试范围、上线阻断权 |

## 团队输出结构

```
zaoren-team-{场景名}/
├── SKILL.md              # 入口：团队概述 + 触发逻辑
├── org-chart.md          # 组织架构图 + 角色关系
├── team-playbook.md      # 团队运行手册（协作协议、决策升级、异常处理）
├── roles/
│   ├── {角色1}.md        # 角色1完整定义（用造人框架生成）
│   ├── {角色2}.md        # 角色2完整定义
│   ├── {角色3}.md        # 角色3完整定义
│   └── ...
└── scenarios/
    ├── daily-ops.md      # 日常运作场景示例
    ├── crisis.md         # 危机处理场景示例
    └── decision.md       # 重大决策场景示例
```

## 团队运行模式

团队生成后，用户与团队的交互有三种模式：

### 模式一：点对点

用户直接跟某个角色对话。适合日常执行。
- 「研究员，帮我看一下这只ETF」
- 「厨师长，新菜单试一下酸菜鱼」

### 模式二：团队会议

用户召集全员或部分角色讨论。自动调用 zaoren-roundtable 的讨论协议。
- 「开个会，讨论下个月的投资策略」
- 「把领班和厨师长叫来，客诉的事聊一下」

### 模式三：自动协作

某些流程由角色之间自动串联，用户只看最终输出和需要决策的节点。
- 研究员出报告 → 基金经理评估组合影响 → 风控官检查风险 → 提交老板决策
- 编辑出稿 → 运营适配平台 → 商务匹配变现 → 提交老板审核

**关键设计**：自动协作流程中，任何角色都可以「拉停」——在自己的职责范围内发现问题，直接阻断流程，不需要等老板批准。风控官发现超限就直接否决，不是建议。

## 团队进化机制

### 角色调整

用户可以随时：
- 增加角色：「加一个市场营销的角色」
- 删除角色：「不需要单独的QA了，合并到开发里」
- 修改角色：「风控官太保守了，调整一下红线参数」

### 协作优化

使用过程中，如果发现：
- 某个环节经常出问题 → 检查是否有角色真空
- 两个角色经常冲突但不产生价值 → 调整边界或合并
- 某个角色几乎不被使用 → 可能是多余的，建议精简

### 从模板到定制

模板只是起点。用户可以上传自己的素材（行业经验、个人方法论、历史决策记录），逐步把模板角色替换为基于真实素材造出的定制角色。

模板角色是「通用底板」，定制角色是「专属大脑」。最终目标是整个团队都是从用户自己的认知体系中长出来的。
