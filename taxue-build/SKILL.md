# taxue-build：系统构建

> 把一次性的解法，变成可复用的资产。

---

## 模式选择

用户输入 `/build` 后，**必须首先询问模式**：

```
你想构建什么类型的工具？

A. 从解法编译 → 你有解决某类问题的方法/流程，想做成可复用模板
   示例：简历诊断流程、周报模板、检查清单

B. 从人物编译 → 你想提取某人的决策逻辑，创造一个可进入的"世界"
   示例：老板的判断方式、某投资人的分析框架
```

---

## 模式A：从解法编译（原taxue-build）

适用场景：个人模板、SOP、检查清单，15分钟完成。

### 六步编译法

```
① 目的：这个 Skill 解决什么问题？
② 标准：什么叫「做好了」？
③ 边界：什么时候不用？
④ 界面：用户怎么调用？触发词、输入、输出。
⑤ 验收：测3个真实场景，通过才算完成。
⑥ 迭代：用起来，根据反馈改。
```

### 输出格式

```yaml
---
name: skill名称
description: |
  一句话说明用途。触发：关键词
---

## 触发条件
## 执行步骤
## 输出格式
```

## 模式B：从人物编译（原zaoren-compile）

适用场景：多角色认知系统、完整Skill，需要设计世界观、角色、盲区。

### Step 1：识别知识类型

| 类型 | 特征 | 来源 | 架构 |
|:---|:---|:---|:---|
| **流程型** | 有明确步骤顺序 | SOP、操作手册 | 工作流步骤为主干 |
| **判断型** | 核心是做决策的框架 | 投资分析、选题标准 | 判断树/决策矩阵 |
| **风格型** | 核心是表达风格 | 个人写作风格、品牌文案 | 风格规则+反例 |
| **知识型** | 大量领域知识 | 书摘、研究报告 | 知识检索框架 |
| **思维模型型** | 隐性框架、决策模式 | 专家分析结构 | 触发条件+调用协议 |
| **复合型** | 兼具以上多种 | - | 识别主类型，其余降级 |

### Step 2：提取可执行核心

- **流程型** → 提取「步骤骨架」+ 输入/输出/决策点
- **判断型** → 提取「判断框架」+ 维度/标准/案例
- **风格型** → 提取「风格DNA」+ 规则/句式/正例反例
- **知识型** → 提取「检索框架」+ 分类/问题类型/索引
- **思维模型型** → 提取「调用协议」+ 触发情境/操作步骤/边界

### Step 3：设计分层结构

```
SKILL.md正文        骨架层：工作流/判断框架/风格DNA（200-300行）
references/         知识层：选项列表/模板/案例/详细规范（200-400行/文件）
运行时输入          动态层：用户每次提供的具体内容
```

> 去掉这段内容，skill还能运行吗？能运行 → 放references；不能运行 → 留SKILL.md

### Step 4：选择文件结构

| 结构 | 适用场景 |
|:---|:---|
| **极简型** | 单一流程，规则少。只有SKILL.md |
| **标准型** | 最常见。SKILL.md + references/options.md + templates.md |
| **领域分支型** | 同一工作流在不同领域差异大。references/domain-a.md... |
| **批量输出型** | 生成多个结构类似的产物。references/unit-templates.md |
| **知识库型** | 知识检索。references/index.md + topic-a.md... |
| **思维模型型** | E类输入。references/frameworks.md + cases.md |

### Step 5：写 SKILL.md

**Frontmatter**（80-120字，宁啰嗦也要覆盖模糊表达）：

```yaml
---
name: skill-name
description: |
  [做什么，一句话]。
  触发条件：[3-5个用户可能说的短语]。
  即使用户只说「[最模糊的触发表达]」也应触发。
---
```

**核心约束章节**（所有类型都需要）：
- 把「必须/不得/重中之重」提炼为编号清单
- 动词开头，每条一行，最多10条
- 越具体越好——「保持风格一致」不如「所有标题用无衬线字体」

### Step 6：自检

- [ ] description覆盖了模糊触发表达
- [ ] SKILL.md里没有大段候选项（>8条的列表应在references）
- [ ] 每个reference文件开头说明「何时读/包含什么」
- [ ] 动态输入没有硬编码
- [ ] 核心约束是具体可操作的，不是抽象原则
- [ ] 去掉SKILL.md任何一段，skill是否仍能运行？

## 边界

- **不做**：从零设计全新skill（用skill-creator）
- **不做**：评估quality、运行测试、迭代优化（用skill-creator）
- **不做**：执行skill描述的工作（安装后直接触发对应skill）
- **不做**：轻量个人工具 → 用模式A；复杂认知系统 → 用模式B

---

## 与兄弟skill的分工

| 需求 | 选择 | 说明 |
|:---|:---|:---|
| 个人模板、SOP、检查清单 | `taxue-build` 模式A | 15分钟完成，自用为主 |
| 多角色认知系统、完整Skill | `taxue-build` 模式B | 需要设计世界观、角色、盲区 |
| 不确定选哪个模式 | 先模式A，复杂了再升级 | 从简单开始 |
