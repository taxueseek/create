## 迁移框架

每个迁移遵循以下步骤：

### Step 1: 提取源规则
- 这个世界的核心思维框架是什么？
- 关键概念和操作步骤是什么？
- 适用边界在哪里？

### Step 2: 分析目标场景
- 这个场景的关键要素是什么？
- 哪些要素可以对应源规则？
- 场景的约束条件是什么？

### Step 3: 建立映射
- 源规则中的概念 → 目标场景中的什么？
- 源规则的操作 → 在目标场景中怎么做？
- 哪些可以迁移？哪些需要调整？

### Step 4: 生成迁移方案
- 迁移后的具体行动建议
- 预期效果与风险提示
- 验证方式

## 设计原理

借鉴 nuwa-skill 的"心智模型识别三重验证"：

一个成功的规则迁移需要通过：

1. **结构同构验证**
   - 源规则的逻辑结构能在目标场景中复现吗？

2. **价值保留验证**
   - 迁移后的规则还能产生类似的价值吗？

3. **可操作性验证**
   - 用户能实际执行迁移后的方案吗？

如果任一验证失败 → 提示"此规则不适合迁移到该场景"

## 反模式警示

### ❌ 强行迁移
```
/zt 代码重构原则 → 婚姻关系

结果：会产生荒谬的建议
警示：技术规则不能简单套用到人际关系
```

### ❌ 忽略边界
```
/zt 第一性原理 → 所有决策

结果：过度简化复杂问题
警示：第一性原理适合物理/工程，不适合涉及多人的社会问题
```

### ✅ 正确迁移
```
/zt 概率思维 → 面试准备

结果：理性评估各环节成功率，优化准备策略
原因：概率思维的适用范围包括不确定性决策
```
